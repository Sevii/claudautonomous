Securely Deploying Hermes Agent

A hardened setup guide for DigitalOcean · OpenRouter

Hermes Agent is an open-source (MIT) autonomous agent from Nous Research that runs on your infrastructure, has persistent memory, writes its own skills, can browse the web, and — critically — executes shell commands and code. It connects to messaging gateways (Telegram / Discord / Slack / etc.) and exposes an optional web dashboard plus an OpenAI-compatible API server.

Two facts drive every security decision below:

  1. It runs arbitrary commands, so the host it touches should be treated as disposable and untrusted.
  2. It listens on the network (gateway + dashboard + API), so inbound exposure must be locked down.

Recommended architecture

One dedicated DigitalOcean droplet → Hermes inside Docker → OpenRouter as the model provider → no inbound ports except SSH. Talk to it via a messaging gateway (outbound) or an SSH tunnel to the dashboard. This way, even if the agent goes off the rails, it is confined to a container on a throwaway VM.

1. Droplet sizing & base hardening

  • Size: Ubuntu 24.04 LTS. 4 GB RAM / 2 vCPU / 20 GB+ disk if you use browser automation (Playwright/Chromium needs ~2 GB on its own). 2 GB / 10 GB is the bare minimum without browsing.
  • Non-root user: create a hermes user; never run the agent as root.
  • SSH: key-only auth, disable password login (PasswordAuthentication no), ideally limit or change the SSH port.
  • DigitalOcean Cloud Firewall: inbound = SSH (22) from your IP only; deny everything else. Leave outbound open (it must reach OpenRouter + messaging APIs). Enable host ufw as defense-in-depth.
  • Enable automatic security updates; keep DO snapshots/backups so you can nuke-and-restore.

2. Run Hermes in Docker (the sandbox)

The official image deprivileges to a non-root hermes user (UID 10000) and keeps all state in a single mounted directory at /opt/data. Run it with resource limits and auto-restart: 

docker run -d --name hermes \
  --restart unless-stopped \
  --memory=4g --cpus=2 \
  -v ~/.hermes:/opt/data \
  nousresearch/hermes-agent gateway run

Because the whole agent lives in this container, its terminal.backend: local commands execute inside the container, isolated from the droplet host. For an extra layer you can set terminal.backend: docker, but nested Docker on a dedicated VM is usually overkill.

3. OpenRouter as provider (secrets handling)

Get a key at openrouter.ai/keys, then keep it in the mounted .env — never in config.yaml, never committed: 

hermes config set OPENROUTER_API_KEY sk-or-...        # writes to ~/.hermes/.env
hermes config set model.provider openrouter
hermes config set model ~anthropic/claude-sonnet-latest
  • The model must support ≥ 64K context or it is rejected at startup.
  • For CI / secrets-manager flows, pass -e OPENROUTER_API_KEY=... on docker run to override .env and keep keys off disk.
  • On OpenRouter, set a credit limit / spend cap on the key — an autonomous agent in a loop can burn budget fast. Use a cheaper auxiliary model for compression, vision, titles, and web summarization to save cost.

4. Network exposure — keep it shut

From the docs: “Opening any port on an internet-facing machine is a security risk. You should not do it unless you understand the risks.” The dashboard and API server bind to 127.0.0.1 by default — leave it that way.

To reach the dashboard, use an SSH tunnel from your laptop instead of exposing it:

ssh -L 8642:127.0.0.1:8642 hermes@<droplet-ip>

If you must expose the API server, require a key and front it with TLS:

docker run ... -e API_SERVER_KEY="$(openssl rand -hex 32)" ...
  • Never set HERMES_DASHBOARD_HOST=0.0.0.0 raw on the open internet — put a TLS reverse proxy (Caddy / nginx) in front.
  • Prefer reaching it through a messaging gateway (Telegram bot, etc.), which is outbound-only and needs no open inbound port.

5. Agent guardrails (config.yaml)

security:
  redact_secrets: true       # mask API-key patterns in logs
  tirith_enabled: true       # scan commands for dangerous ops
  approvals:
    mode: manual             # manual | smart | off  -> never "off" outside a sandbox

Start with approvals: manual so you confirm flagged commands; relax to smart only once you trust it. Disable toolsets you do not need (agent.disabled_toolsets: [web, ...]) to shrink the attack surface.

6. Run as a service

Docker --restart unless-stopped (or a Compose file with deploy.resources.limits) gives supervised restarts via s6-overlay as PID 1. 

services:
  hermes:
    image: nousresearch/hermes-agent:latest
    container_name: hermes
    restart: unless-stopped
    command: gateway run
    volumes:
      - ~/.hermes:/opt/data
    deploy:
      resources:
        limits:
          memory: 4G
          cpus: "2.0"
Never point two gateway containers at the same ~/.hermes directory — session files are not concurrent-write safe. Use separate dirs + ports per profile.

Quick checklist

  • Dedicated, disposable droplet; non-root user; key-only SSH
  • DO firewall: inbound SSH-from-my-IP only
  • Hermes in Docker with mem + cpu limits and a restart policy
  • OpenRouter key in .env; spend cap set on the key
  • Dashboard / API on 127.0.0.1; reach via SSH tunnel or messaging gateway
  • approvals: manual, tirith_enabled: true, redact_secrets: true
